# 変更要求（CR）ログ

変更要求はGoogle スプレッドシートで管理しています。

📄 **[CRログ（スプレッドシート）]({{CR_LOG_SPREADSHEET_URL}})**


## ルール

### 1) 「採用＝確定」の条件

以下が揃ったら「採用」にする（＝実装確定）。

- 影響（工数/納期/費用）が記入されている
- 承認者が記入されている
- 反映先（仕様書の章 / 画面 / 帳票）が特定できている
- スコープ/納期を固定する場合は、入れ替え（後述）が決まっている

### 2) スコープ/納期固定のための原則（トレードオフ）

追加要望は、基本的に「何かと入れ替え」になる。

- 原則：採用するなら、削る/後ろ倒す/次フェーズ送りをセットで決める（one-in-one-out）
- 入れ替えが決められない場合：一旦「見送り（次フェーズ候補）」として残す

### 3) 承認の基準

- 番頭さん承認で着手（例）
  - 文言修正・軽微な表示調整
  - 既存項目の並び順/ラベル変更
  - 影響が小さい（目安：〜1〜2時間）
- 社長承認で着手（例）
  - 納期/費用が動く可能性がある
  - DB（スプレッドシート構造）変更が必要
  - 請求/給与/外注費など金銭ロジックに影響する
  - 帳票（PDF/Excel）の仕様追加/変更
  - 影響が大きい（目安：0.5日以上）

### 4) 同時進行（WIP）制限

「採用」が増えすぎると開発が止まり、納期が守れなくなる。

- 「採用（未完了）」の上限を決める（例：同時に最大2件まで）
- 上限を超えるCRは「見送り（次フェーズ候補）」に積む

### 5) 完了（Done）の定義

「完了」は実装だけでなく、ドキュメント・リンクまで揃っている状態とする。

- 仕様書（v1.3）または実装仕様（docs配下）の反映箇所リンクがある
- 実装（チケット/PR/コミット）のリンクがある
- 影響範囲の動作確認が終わっている（最低限：対象画面/帳票）

### 6) 書き方ルール

検討コストを下げるため、起票時点で「現状」と「期待」を最低限書く。

- 現状：今どうなっているか（スクショ/例があると早い）
- 期待：どうなってほしいか（受入条件：OK/NGが判断できる形）
- 目的：なぜ必要か（ミス削減、作業時間短縮、顧客対応など）



## 運用ルール

1. **起票** → 追加要望が出たらスプレッドシートに1行追加
2. **影響確認** → 工数/納期/費用を記入
3. **承認** → 顧客と協議し、ステータスを更新
4. **反映** → 採用されたら仕様書・実装に反映、リンクを記載

## ステータス

| ステータス | 意味             |
| ---------- | ---------------- |
| 起票       | 要望を受け付けた |
| 検討中     | 影響を調査中     |
| 採用       | 実装する         |
| 見送り     | 今回は実装しない |
| 完了       | 実装・反映済み   |
