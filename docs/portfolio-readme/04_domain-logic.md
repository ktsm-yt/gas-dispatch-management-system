# 04. ドメインロジック実装

建設労務派遣業固有の業務ルールをどう設計・実装したか。

---

## 1. 建設労務派遣業の特殊性

一般的な勤怠管理・請求システムとは異なる、業界固有の複雑さがある。

- **日雇い・日替わりの現場配置**: 固定シフトではなく、毎日異なる現場・スタッフ構成が変わる
- **1日の中に複数の時間区分**: 上棟 / 終日 / AM / PM / 夜勤 / 未定 の6区分
- **顧客ごとに異なる単価体系**: 基本 / 鳶 / 荷揚げ / 鳶荷揚げ / 半日 / 終日 / 夜勤 / 休日の8種
- **請求書フォーマットが顧客ごとに異なる**: 4種類のフォーマット（format1〜3 + 頭紙）
- **顧客ごとに異なる支払サイクル**: 月末締め・翌月払い等の個別設定
- **源泉徴収の適用**: 甲欄日額表に基づく計算（自社・学生スタッフのみ）
- **人工割（にんくわり）**: 不足人員分の日当を残りのスタッフに案分する独自計算

## 2. 単価区分の設計 — 「表示」と「計算」の分離

最も設計に悩んだ部分。「ダッシュボードの表示列」と「単価計算の区分」は別の概念。

| カラム | 用途 | 値の例 |
|--------|------|--------|
| `display_time_slot` | ダッシュボード表示列 | am, pm, night, jotou, shuujitsu, mitei |
| `pay_unit` | スタッフ給与計算 | basic, half, fullday, yakin, tobi, tobiage |
| `invoice_unit` | 顧客請求計算 | **pay_unit と常に同値** |

`pay_unit` と `invoice_unit` はスキーマ上は別カラムだが、`AssignmentService` で常に同値に強制される。請求単価と支払単価にズレがあると不正マージンの温床になるため、搾取防止の業務ルールとして同期を強制している。

```
[ダッシュボード: AM列に表示]
        ↓
display_time_slot = "am"
pay_unit = "tobi"        → スタッフには鳶単価で支払い
invoice_unit = "tobi"    → 顧客にも鳶単価で請求（pay_unit と常に一致）
```

**なぜ `display_time_slot` を分離するか**: 表示列と単価区分は1:1対応しない。AM列に表示される配置でも、単価区分は `basic`（基本）や `tobi`（鳶）など業務実態に応じて異なる。また、同じAM列に `half`（半日単価）の配置が2件入るケースもある。表示と計算を分離することで、ダッシュボードの見た目を変えずに単価体系を柔軟に運用できる。

**なぜカラムを2つ持つか（pay_unit / invoice_unit）**: 現在は同値を強制しているが、スキーマとしては分離を維持している。将来的に分離が必要になった場合のスキーマ変更を回避するため。

## 3. 人工割（にんくわり）計算

建設業独自の概念。5人必要な現場に4人しか行けなかった場合、残りの4人の日当を割増する。逆に過剰配置なら減額する。

### 単純な案分ではない

納期4日前にクライアントから「人工割ってどうやるの？」と言われて発覚した。仕様書にも過去の会議録にも一切記載がなく、ヒアリング時にも共有されなかった根幹仕様だった。さらに、いざ仕様を確認すると計算ルール自体がクライアント固有かつ曖昧で、単純な案分では済まなかった。

仕様を確認するために質問書を作成したが、回答にも曖昧さや矛盾が多く、7項目の追加確認書を期限付き（暫定方針明記）で再送する必要があった。

**クライアント回答で判明した曖昧さの例**:

| 確認事項 | 曖昧だった点 |
|----------|-------------|
| 係数の根拠 | 「1.5でつけておいて」と指示されたが、自動計算（required÷actual）だと4.0になるケース。手動指定なのか例外なのか不明 |
| 外注スタッフ | type 4（一人親方）は「減額する」、type 5（外注先）は「よく理解できませんでした」と未回答 |
| 請求側への適用 | 「ABが正解」と回答されたが、AとBは排他的な選択肢だった |
| 延長・残業への適用 | 「全て適用」と回答されたが、そもそもデータが存在しない項目だった |

暫定方針を示し、未回答なら暫定方針で実装＋後日修正は有償と明記して回答を引き出した。

### 最終的な実装

```typescript
// calc_utils.ts — 人工割係数（0.1刻み切捨て）
function calculateNinkuCoefficient_(required: number, actual: number): number {
  if (required <= 0 || actual <= 0) return 1.0;
  if (required === actual) return 1.0;
  return Math.floor((required / actual) * 10) / 10;
}

// 例: 5人必要・4人配置 → 1.2（割増）
// 例: 4人必要・5人配置 → 0.8（減額）
```

**切り捨てによる差額消滅**: 当初は正確な係数（小数点以下制限なし）で実装していたが、クライアントから「0.1刻みにしてほしい」と要望があった。0.1刻み切捨てにすると分配元と分配先の合計が一致しなくなる — 例えば日当12,000円・係数1.3の場合、`12,000 × 1.3 = 15,600円 × 3人 = 46,800円` だが、4人分の満額は48,000円。差額1,200円は会社側の得になる。この差額の発生を具体的な金額例で説明し、了承を得た上で実装した。

**適用範囲の判定ロジック**:
- 外注スタッフ（type 5）は人工割の対象外（契約金額固定）
- 外注スタッフも actual_count にはカウントする（係数計算に影響）
- 交通費・手動調整額は人工割の対象外
- 人工割調整額は源泉徴収の対象（日額テーブル参照時に調整込みの金額で計算）
- 配置変更時は同現場の全スタッフを再計算（係数が全員に波及するため）

**学び**: クライアントにとって「当たり前すぎて言語化されない業務知識」が、給与計算の根幹を左右することがある。共有されなかった仕様に対しては、曖昧な回答をそのまま実装せず、具体的な金額例で選択肢を提示し、期限と暫定方針を明記して仕様を確定に持ち込む対応が不可欠だった。

## 4. 帳票出力の複雑さ

### 4種類のフォーマット

| フォーマット | 特徴 | 実装難易度 |
|-------------|------|-----------|
| format1 | セル結合多数、ヘッダー繰り返し | ★★★★ |
| format2 | format1 + 注文番号・支店の分離カラム | ★★★★★ |
| format3 | 非結合、最もシンプル | ★★ |
| atamagami（表紙） | セル結合多数、集計ページ、封筒用レイアウト | ★★★ |

### Driveテンプレートアプローチ

```
1. Drive上のテンプレート（Google Sheets）を makeCopy()
2. コピーに固定セル（顧客名・期間等）を書き込み
3. 明細行をループで書き込み
4. SpreadsheetApp.flush() で確定
5. PDF/Excel にエクスポート
```

**最大の壁**: Google SheetsにはPageBreak APIがない。動的に生成されるデータシートの明細行数に応じて体裁を整える必要があるが、印刷時の改ページ制御ができない。スプレッドシートを中間生成物として利用し、印刷レイアウト（テンプレート）とデータ入力を別シートに分離、複数シートを合体させてPDF化する実装に落ち着いた。

丸3日をテンプレート再現に費やした。1セルずつの調整、文字切れ対処、ページ区切り境界の調査。1mmのズレも許されない調整作業。

```
[年度末: 2月締め]
        ↓
[6月1日 自動アーカイブ実行]
  ├── T_Jobs, T_JobAssignments → 別スプレッドシートへ移行
  ├── T_Invoices, T_InvoiceLines → 別スプレッドシートへ移行
  └── T_Payouts → 別スプレッドシートへ移行

[M_* マスター、T_MonthlyStats] → 永続保持（年次比較分析用）
```

**透過的読み取り**:

```javascript
// includeArchive オプションでアーカイブDBも検索
const results = searchInvoices({ customerId, yearMonth, includeArchive: true });
// → 現在のDBとアーカイブDBを統合して返却
// アーカイブ元のレコードには _archived: true フラグ付与
```

**Pre-archive safeguard**: 5月15日に警告メール送信、5月15〜31日は猶予期間（請求書・給与の最終処理）、6月1日に自動実行。

## 6. 源泉徴収計算

国税庁の令和7年分 給与所得の源泉徴収税額表（日額表・甲欄・扶養0人）をTypeScriptの定数配列としてコードに埋め込み、二分探索でO(log n)ルックアップする仕組み。24,000円超は累進計算式で算出する。

```typescript
// WithholdingTaxTable.ts — 税額テーブル（国税庁データをコード埋め込み）
const WITHHOLDING_TAX_TABLE_R7: readonly [number, number][] = [
  [0, 0], [2900, 5], [3000, 10], /* ... 230エントリ ... */ [23900, 2295],
];

// 二分探索で税額を取得
function lookupDailyWithholdingTax(dailyAmount: number): number {
  if (dailyAmount >= 24000) { /* 累進計算式 */ }
  // 23,900円以下: テーブルを二分探索
}
```

**計算単位: 1日合算ではなく配置（現場）単位**: 当初は1日の給与を合算してからテーブル参照していたが、日額表は累進構造のため合算すると税額が過大になる。例えばAM現場8,000円＋PM現場8,000円を合算すると16,000円に対する税額（890円）が適用されるが、配置単位で個別参照すれば8,000円×2回（210円×2＝420円）で済む。実運用でこの過大徴収が発覚し、配置単位の個別参照に修正した。

**適用条件**:
- `M_Staff.withholding_tax_applicable` フラグで制御
- 自社スタッフ（regular）と学生（student）に適用（給与所得）
- 個人事業主（sole_proprietor）・外注（subcontract）には適用しない（事業所得）
