# 06. プロジェクトマネジメントと成長記録

## A. プロジェクトマネジメント実績

### 開発タイムライン

| フェーズ | 期間 | 内容 | 判断ポイント |
|----------|------|------|-------------|
| Phase 1（コア） | 約2ヶ月 | 案件管理・配置ダッシュボード・マスター管理 | GAS選定、3層アーキテクチャ採用 |
| Phase 2（請求・給与） | 約2ヶ月 | 請求書生成・給与計算・帳票出力 | テンプレートアプローチ確定、TS移行開始 |

### AI分業モデルの設計思想

PM/PO・UXを自分が担当し、実装をAIに委譲する体制をとった。これはリソース配分の最適化であると同時に、「人間にしかできない判断」と「AIが得意な作業」の明確な分離だった。

| 担当 | 役割 | 理由 |
|------|------|------|
| 人間 | 要件定義・UI判断・ステークホルダー対応 | 業務理解・感情理解が必要 |
| AI（実装） | コード実装・テスト生成 | 正確性・速度が強み |
| AI（レビュー） | 設計レビュー | 実装者と異なる視点でレビューバイアスを排除 |

この分業により、未経験エンジニアが **79,000行・210PR規模** のプロジェクトをソロで完遂した。

### スコープクリープとの戦い

#### 1. 見えない努力の可視化

フィードバックが少ない環境で、仕様書の穴を埋めるべく独自に設計・実装を重ねていた（当時49項目・AI推定283時間分、現在は56項目に増加）。しかしクライアントにはこの「見えない努力」が見えていなかった。

クライアントからの変更要求が仕様書の有償/無償ラインを越え始めた時、問題の核心に気づいた。

> 「見えない努力」を見えないまま抱えていること。

#### 2. 「数字の暴力」による交渉戦略

AIにプロジェクトを走査させ、仕様書外の独自実装を全件洗い出した。

| 項目 | 内容 |
|------|------|
| 独自設計 | 49項目（交渉時点。現在は56項目） |
| AI推定工数 | 283時間（交渉時点の見積） |
| 内訳 | 新機能12件、UX/UI設計23件、機能拡張8件、技術判断8件 |

このドキュメントを、クライアントの社内検討が結論に至る前に先手で提出した。

> 圧倒的な数字で「見えない努力」を可視化し、傾いた認識のオセロをひっくり返す。

**結果**: クライアントは態度を改め、以降の変更要求を有償対応に切り替えることに合意。

#### 3. 変更要求の有償化 → 単価2倍改定

検収期間（本来「仕様通りに動くか確認する期間」）に約50件の追加要望が殺到した。中にはDB構造変更やテスト全体やり直しを伴う大規模改修も含まれていた。

冷静に以下を整理してクライアントに提示した。

- 検収期間の定義
- プロジェクトの目的
- 仕様書の有償・無償の定義
- これまでの善意対応の総量

**結果**: 追加開発として切り分け、**開発単価の2倍改定**を獲得。2週間でPR 64本を処理し、最終的に「保守を続けてほしい」と依頼される関係に発展した。

### クリティカルな仕様変更への対処（人工割事件）

**状況**: 納期4日前、計算の根幹を揺るがす「人工割」（建設業独自の日当案分計算）が仕様に含まれていなかったことが発覚。仕様書にも過去の会議録にも一切記載がなかった。

> 「やぁ、当たり前にできるもんだと思ってた」——クライアントの一言

**意思決定プロセス**:

1. 即座にQ&Aシートで詳細仕様を質問（案分率、端数処理、スタッフ種別の切り分け等）
2. 1週間返答なし → 連休挟み → 納期4日前に不明確な回答が到着
3. 感情コントロール: AIに経緯を投げ、共感と正当性の確認を求めた後、冷静な対応策を立案
4. AorBのQ&A方式で仕様を再提示。回答期限と暫定方針を通告:
   - 「安全に出せる範囲」を凍結
   - 未確定仕様は納品後に先送り
   - 期限までに回答がなければ暫定方針で確定

**学び**:

- 業務の「当たり前すぎて言語化されない暗黙知」が最大のリスク
- 感情的になった時こそAIに壁打ちし、冷静な判断材料を整理する
- 二択（AorB）形式の質問は曖昧な回答を防ぐ有効な手法

### ステークホルダーマネジメント

**フィードバックなし環境への適応**

最速でデモ版を共有しても1週間フィードバックなし。Yes/Noの質問でも電話で説明を求められる状況が続いた。非同期コミュニケーションが機能しない相手に対しては、選択肢を絞り込んだ提示と「このまま進めます」という通告型コミュニケーションが効果的だった。

**プロダクトオーナー的意思決定**

具体的な指示がほぼないため、自らがPOとして仕様を決定し続けた。「こういう仕様にしてしまおう」と自由に作れるメリットがある一方、常に不安がつきまとう。この経験から、明確な要件定義フェーズとステークホルダーとの仕様合意プロセスの価値を強く実感した。

**保守契約への発展**

当初予定していなかった保守契約を獲得。「案件管理のミスがなくなった」「毎日かけていた請求書作業がボタンひとつで完成する」というクライアントの声が、プロダクトの価値を示している。

## B. 技術的反省と成長

### 技術選定の不可逆性

> 「技術選定は、プロジェクトのその後の全てを決める」

GASを選んだ時点で決まっていたこと:

- E2Eテストの自動化は事実上不可能（OAuth BOT検知、iframe制約）
- 複数シート横断処理は数秒かかる（I/O遅延の物理的限界）
- これらへの対処に余計な工数を取られ、開発効率が低下する局面があった

初期設計時にはこの規模になるとは想定していなかった。契約条件への配慮から技術選定の後からの修正ができなかった。ビジネスサイドとエンジニアリングサイドの双方を経験して身にしみた教訓。

---

### JS→TS移行の経験と反省

- Phase 1でTypeScript導入を見送った判断は反省点
- 型安全のないJavaScriptでの開発は、規模が大きくなるほど恐怖が増す
- 25,000行規模のJS→TS移行を実施。GAS独自の制約（コンパイル挙動等）の把握不足が一因
- 人工割事件の直前にバックエンドのTS化を進めていたことで、納期4日前の緊急対応が救われた

**教訓**: 「後でリファクタする」は誘惑だが、スケールした後の移行コストは初期導入コストの何倍にもなる。

---

### テスト設計の変化

開発初期は「まず動くものを」という方針でテストは後手に回っていた。

**AI開発でのテストの重要性**:

- AIはテストを書かずに実装すると、「動いているように見えるが壊れているコード」を生成することがある
- テストを先に書き、テストを通過するように実装させるのがAI開発のベストプラクティス
- AIが「必ず通るテスト」を意図的に書くという問題（アサーション偽装）を発見
- 対策としてAIチート検出を `/review` スキルに統合し、開発フローに組み込んだ

### 現時点の課題認識

正直に認識している不足:

| 課題 | 詳細 |
|------|------|
| CS基礎 | アルゴリズム・データ構造の体系的理解 |
| モダンスタック | React + RDB等での実装経験 |
| チーム開発 | コードレビュー文化、マージ戦略、CI/CDの本格運用 |

これらはソロ開発・GASという制約の中で得難かった経験。次のフェーズで積極的に補いたい領域。

## C. 私が提供できる価値

### 1. 現場の言葉 → 技術実装への翻訳

建設業に関わる立場だったため、鳶・上棟といった業務用語や現場の肌感覚を持った上でシステム設計に臨めた。ドメイン知識のない技術者が作ると「計算は合っているが現場で使えない」プロダクトになる。この業界理解がプロダクトの品質を支えた。

一方で、「人工割」のように引き出しきれなかった暗黙知もある。その反省から、AorB形式の質問設計、Q&Aシートによる仕様確認、回答期限付きコミュニケーションといった仕組みで対処した。最初から完璧に聞き出せるわけではないが、**失敗から仕組みを作って再発を防ぐサイクル**を回せることが、業界を問わず持ち込める自分の強みだと考えている。

---

### 2. ビジネス要件とエンジニアリングの「共通言語を作る」能力

AI活用で得た本質的な学び:

> 概念を伝えれば、AIはより良い手段を自ら考え整えてくれる。AIはハンドリングする人間の質が重要。

「SWR」「Read-Once最適化」「楽観的ロック」といった上位概念を知ることで、AIへの指示精度が劇的に向上した経験がある。技術概念のインデックスを持つことの価値を体感した。

これはPM/POがエンジニアと話す時にも同じことが言える。完全な実装知識は不要でも、概念レベルで共通言語を持てる人間が、チームのスループットを上げる。

---

### 3. スコープの地雷を早期発見・円滑に処理するスキル

- 独自実装（当時49項目、現在56項目）を可視化し、交渉材料に転換
- 変更要求の有償化、単価2倍改定を獲得
- 感情的になる場面でもAIを活用して冷静な判断を維持

スコープクリープは多くのプロジェクトが失敗する主因のひとつ。「見えない努力を数字にして示す」という手法は、チームの防衛にも応用できる。

---

### 4. AI開発チームのマネジメント経験

複数のAIを実装・レビュー・調査・チート検出など適材適所で使い分け、ソロで79,000行・210PR規模のプロジェクトを完遂した。

AI時代のAI開発チーム編成を実践的に経験した人間として、「AIをどう使うか」ではなく「AIとどう協働するか」という問いに対して、実例から話せる。

---

## 最後に

> 仕様を待ち、言われた通りのプロダクトを納品するのではなく、自ら率先して働きかけ、技術的な制約とビジネス価値を天秤にかけて仕様を決定する。

> スコープの地雷を円滑に処理し、チームが開発に集中できる環境を作る——橋渡しとしての力でチームに貢献する。

このプロジェクトを通じて、PM/POは「管理する人」ではなく「翻訳する人」「防衛する人」「決断する人」だと体感した。技術的な深さを持ちながら、ビジネスの文脈で判断できる人材として、次のフィールドでその力を発揮したいと考えている。
