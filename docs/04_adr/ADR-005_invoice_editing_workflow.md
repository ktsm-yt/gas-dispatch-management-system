# ADR-005: 請求書編集ワークフローと運用フロー設計

## ステータス

承認待ち（要件ヒアリング中）

## コンテキスト

P2-2請求書機能において、以下の検討が必要：

1. **請求書の編集可能範囲の定義**
   - どのフィールドが編集可能か
   - どの状態で編集を許可するか
   - 元データ（Job, Assignment）への参照の維持方法

2. **ステータス体系の統一**
   - 旧ステータス: `draft`, `issued`, `unsent`, `sent`, `paid`, `unpaid`
   - 現在の設計では: `draft`/`issued` → `unsent`への統合検討

3. **運用フローの不確実性**
   - 番頭さんによる承認フロー
   - 社長によるチェックフロー
   - 実装スコープの判定が必要

## 決定

以下の3つの選択肢を提示し、実際の運用フローをヒアリング後に決定する：

### 選択肢A: シンプル編集モデル
- **unsent状態のみ編集可能**
- 単価、数量、摘要（金額は自動計算）の編集を許可
- 送付後（sent以降）は完全にロック
- 編集制約: `job_id`, `assignment_id` は常に編集不可

### 選択肢B: 段階的承認モデル
- draft/unsent: 自由編集可能
- issued: 番頭が確認・編集可能
- sent: 社長が確認・コメント可能（編集不可）
- paid/unpaid: ロック

### 選択肢C: ハイブリッドモデル
- unsent: 自由編集
- issued: 特定フィールドのみ編集（金額部分など）
- sent以降: 完全ロック

## 理由

1. **運用不確実性への対応**: 番頭→社長のダブルチェックフローが不明なため、先に仕様を定義してからの実装が重要
2. **バックエンド完成済み**: InvoiceService.saveは複数フィールド編集対応済み
3. **フロントエンド未実装**: UI実装時点で最終決定をするアプローチが可能

## ステータス統一の設計

```
旧体系              新体系
draft    --|
issued   --|-----> unsent
unsent   --|

sent                sent
paid                paid
unpaid              unpaid
```

**後方互換性**: migrate時に旧ステータスを新ステータスに変換
**遷移ルール**:
- `unsent` → `sent` (送付時)
- `sent` → `paid` (支払済み) または `unpaid` (未払い)

## 詳細

### バックエンドの現状

**実装済み**:
- `InvoiceService.save()`: 複数フィールド同時編集（upsert）
- 編集可能フィールド: `amount`, `subtotal`, `tax`, `remarks`（単価/数量を含む詳細は明細側）
- 編集不可フィールド: `job_id`, `assignment_id`（参照整合性維持）

**未実装**:
- フロントエンドUI（請求書編集フォーム）
- ステータス遷移ロジック（draft/issued → unsent への変換）

### ヒアリング項目（待機中）

1. 現在の承認フロー
   - 番頭さんの役割は何か？
   - 社長のチェックポイントは？
   - 複数回の修正が発生するか？

2. 編集可能なタイミング
   - 何時点までは修正OKか？
   - 一度送付したら修正不可か？
   - 再発行（reissue）の仕組みが必要か？

3. 監査・ログ要件
   - 誰が何を修正したかの履歴は必要か？
   - 金額変更の承認フローが必要か？

## 実装スケジュール（案）

1. **運用ヒアリング完了**
   - 選択肢A/B/Cのいずれかに決定
   - ステータス設計の最終化

2. **フロントエンド実装**（KTSM-88）
   - 編集フォーム実装
   - ステータス遷移UI

3. **テスト・リリース**
   - E2Eテスト実装
   - 顧客での動作確認

## 影響

- **フロントエンド開発**: 編集UIのスコープが確定
- **バックエンド**: 既存APIは変更不要（互換性維持）
- **運用**: ステータス遷移の明確化により、ユーザー教育資料の作成が可能

## 関連する学習ポイント

このADRの検討プロセスから得られた学習：

1. **運用フロー不明時の対処法**
   - 実装に進まずにヒアリング項目を整理することが重要
   - 設計オプション（A/B/C）を先に提示し、顧客の運用パターンを理解してから決定する

2. **バックエンド・フロントエンドの分離**
   - バックエンドは複数フィールド編集対応済み（汎用的に設計）
   - フロントエンドUIの実装時点で、編集制約やステータス遷移を具体化する

3. **後方互換性の維持**
   - 既存ステータス（draft, issued）を許容しながら、新体系（unsent）への移行を段階的に進める

4. **コミット単位の設計**
   - 編集制約の実装、ステータス遷移ロジック、UI実装は独立してコミット可能
   - 各機能変更にテストを含める
