# ADR-005: 請求書一括生成のパフォーマンス制約

## ステータス
承認済み

## コンテキスト
請求書一括生成機能において、53社の処理に約8分かかるという状況がある。お客様から「遅い」と感じられる可能性があるため、技術的な背景を説明できるようにする必要がある。

## 技術的制約

### Google Apps Script (GAS) の制限

1. **スプレッドシートI/Oの遅延**
   - GASはスプレッドシートへの読み書きが1回あたり100ms〜数秒かかる
   - データベースのような高速なクエリ処理ができない
   - 全件読み込み → メモリでフィルタリング という処理パターンが必須

2. **実行時間制限**
   - 1回の実行は最大6分まで
   - そのため10件ずつバッチ処理し、フロントエンドから連続呼び出しする設計

3. **同時実行制限**
   - 同一ユーザーの同時実行数に制限あり
   - 並列処理による高速化は困難

### 実測値（2026年1月時点）

| 項目 | 値 |
|-----|-----|
| 総顧客数 | 53社 |
| 処理時間 | 約8分 |
| 生成件数 | 13件 |
| スキップ件数 | 40件（配置データなし） |
| 1バッチ（10社） | 約80秒 |

### 最適化済みの内容

以下の最適化を実施済み：

1. **バッチI/O集約**
   - 案件・配置・請求書データを各1回だけ読み込み
   - メモリ上でインデックス構築・結合
   - 従来: 10社で約100回のシートI/O → 最適化後: 約5回

2. **一括挿入**
   - 請求書・明細を個別挿入ではなく一括挿入
   - シート書き込み回数を最小化

3. **請求番号の連番処理**
   - ロック取得を1回に集約
   - メモリ上で連番配布

### これ以上の高速化が困難な理由

- スプレッドシートをデータストアとして使用している限り、I/O遅延は避けられない
- Firebaseや外部データベースへの移行は、既存システムとの互換性・運用コストの観点から現実的でない
- GASの実行環境自体の性能向上は制御不可能

## 決定

1. **現状のパフォーマンスを許容する**
   - 53社で8分は、月1回の処理として許容範囲内
   - 自動連続実行により、ユーザーは放置するだけでよい

2. **UX改善で対応**
   - ワンクリックで最後まで自動実行
   - 停止ボタンで中断可能
   - 進捗バーでリアルタイム表示

3. **お客様への説明ポイント**
   - 「Googleスプレッドシートを使用しているため、データベースシステムと比べると処理速度に制約があります」
   - 「月1回の処理なので、開始ボタンを押して放置していただければ自動で完了します」
   - 「処理中でも停止ボタンで安全に中断できます」

## 将来の改善オプション（必要に応じて）

1. **バッチサイズの調整**: 10件→20件に増やして往復回数削減（要テスト）
2. **時間主導トリガー**: サーバーサイドで定期自動実行（ブラウザ不要）
3. **データベース移行**: Firebase/Cloud SQLへの移行（大規模改修）
