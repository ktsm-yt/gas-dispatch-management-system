# ADR-004: GitHub CI/CDを導入しない

## ステータス

承認済み

## コンテキスト

Google Apps Script（GAS）プロジェクトの開発において、以下の選択肢があった：

1. GitHub ActionsでのCI/CD自動化
   - PR時の自動テスト・リント実行
   - 自動デプロイメント
2. ローカルチェック + 手動デプロイ
   - clasp pushによる手動デプロイ
   - 開発者によるローカル確認

## 決定

**GitHub ActionsによるCI/CDは導入せず、ローカルチェック前提の手動デプロイを採用する。**

## 理由

### 1. GASの特性上、CI環境でのテストが困難

- GAS APIの大半はGoogle環境に依存（Spreadsheet、Drive、PropertiesService等）
- ローカルでモック環境を構築するコストが高い
- 実環境でのテストが不可避

### 2. デプロイは手動が適切

- GASデプロイは`clasp push`による手動実行が標準
- 本番/開発環境は同一GASプロジェクト内の異なるデプロイメントで管理
- Google認証が必要なため、自動化のメリットが薄い

### 3. ローカルチェックで品質担保が可能

- `clasp push`実行時に構文エラーが検出される
- GASエディタでの動作確認が必須のため、自然とローカルチェックが行われる
- PRレビュー時に差分を確認

### 4. プロジェクト規模とチーム構成

- ソロまたは小規模チーム開発
- 全てのコミットがレビュー済み
- CIのセットアップ・メンテナンスコストが利益を上回る

### 5. 自動化可能なチェックが限定的

現状、自動化できるチェック：
- `npm run env:check`（環境変数チェック）
- `npm run render:docs`（ドキュメント生成）

これらは開発補助ツールであり、コード品質やバグ防止には直結しない。

## 影響

### プラス

- セットアップ・メンテナンスコストの削減
- CI実行待ち時間の削減
- シンプルな開発フロー維持

### マイナス

- 自動リント・型チェックなし（将来導入する場合は要対応）
- ローカルチェックを忘れるとPRで発覚

## 補足

### TypeScriptについて

GAS環境ではTypeScriptを導入しない方針。理由：
- GASの実行環境はJavaScriptのみ
- 型チェックのメリットよりもトランスパイルのオーバーヘッドが大きい
- データベーススキーマは事前に詳細設計済みのため、型定義の必要性が低い

### データベース設計

スプレッドシートベースのデータベーススキーマは事前に詳細設計済み（`docs/03_spec/05_database.md`参照）。
実装フェーズでスキーマ設計を行う必要はない。

## 将来的な見直しポイント

以下の条件が満たされた場合、CIの導入を再検討する：

1. チーム規模が拡大（3人以上の同時開発）
2. GAS非依存のロジック層が大幅に増加
3. ローカルチェック漏れによる品質問題が頻発
